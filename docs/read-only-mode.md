# Safety modes: `--dry-run` and read-only

`confluence-cli` gives you two independent guardrails before any mutation:

- **`--dry-run`** — preview the HTTP request the command *would* send.
- **Read-only mode** — block every write for the session.

They compose: `CONFLUENCE_CLI_READ_ONLY=1 confluence-cli page delete 123
--yes --dry-run` is a useful, byte-identical preview that cannot
accidentally delete anything.

## `--dry-run`

Every mutating command accepts `--dry-run`. The flag short-circuits the
HTTP call and prints the planned request:

```bash
$ confluence-cli page delete 123 --yes --dry-run
{
  "dry_run": true,
  "method": "DELETE",
  "url": "https://acme.atlassian.net/wiki/rest/api/content/123"
}
```

Commands that accept `--dry-run`:

| Subtree | Commands |
|---|---|
| `page` | `create`, `update`, `delete`, `move`, `copy`, `restore`, `watch`, `unwatch` |
| `comment` | `add`, `update`, `delete` |
| `attachment` | `upload`, `update`, `delete` |
| `label` | `add`, `remove` |

The preview is generated by the same build helper the live write uses,
so the URL and payload cannot drift from what would actually be sent.
Operations that need to read a current version (page update / move /
restore, comment update / delete) still perform the safe GET — the
preview reflects the resulting payload.

`--dry-run` is **not blocked by read-only mode**. It never sends HTTP,
so it is the recommended way to inspect a write without flipping any
switch.

## Read-only mode

Read-only mode is a session-level switch. When active, every mutating
client method is blocked **before** the HTTP request is built.

### Enabling

Either set it in config:

```yaml
# ~/.angelmsger/confluence/config.yaml  (legacy: ~/.confluence/config.yaml)
defaults:
  read_only: true
```

Or via environment for an ad-hoc session:

```bash
export CONFLUENCE_CLI_READ_ONLY=1
```

Environment beats config when both are set.

### Blocked operation example

```bash
$ confluence-cli page delete 123 --yes
{
  "error": {
    "category": "permission",
    "code": "READONLY_BLOCKED",
    "message": "operation \"DeletePage\" blocked: read-only mode is enabled",
    "hint": "Re-run with --allow-writes to permit writes for this invocation, or unset CONFLUENCE_CLI_READ_ONLY / defaults.read_only.",
    "next_steps": [
      "Add --allow-writes to the command line",
      "unset CONFLUENCE_CLI_READ_ONLY",
      "Set defaults.read_only=false in ~/.angelmsger/confluence/config.yaml"
    ],
    "retryable": false
  }
}
# exit code: 5  (category=permission)
```

### Per-invocation override: `--allow-writes`

The root-level `--allow-writes` persistent flag overrides the posture for
exactly one invocation:

```bash
CONFLUENCE_CLI_READ_ONLY=1 confluence-cli --allow-writes page delete 123 --yes
```

This is the only knob; there is no `--read-only` flag. The asymmetry is
deliberate: persistent posture flows from config/env, per-call overrides
flow from a single flag, and the two cannot conflict.

### Scope

| Layer | Blocked? |
|---|---|
| Confluence API writes (POST/PUT/DELETE) | Yes |
| `attachment download --output <path>` (writes remote content to a local file) | No |
| `config init`, `auth login|logout`, `config use-context` | No |
| `skill install / uninstall` | No |

The boundary is "remote Confluence service". `confluence-cli`'s own
configuration is intentionally outside the scope — otherwise an agent
that turned on read-only would lose the ability to turn it back off.

## Cookbook

```bash
# Safe summarise session — read everything, mutate nothing.
export CONFLUENCE_CLI_READ_ONLY=1
confluence-cli search --text "release notes"
confluence-cli page get 123
confluence-cli comment list 123
# A misclick on a write is now a clean 5/READONLY_BLOCKED:
confluence-cli page delete 123 --yes   # blocked, never sent

# Inspect the planned delete without leaving read-only:
confluence-cli page delete 123 --yes --dry-run

# When you do want to send it, one-shot override:
confluence-cli --allow-writes page delete 123 --yes
```
